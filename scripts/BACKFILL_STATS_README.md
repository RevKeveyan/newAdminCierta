# Backfill Statistics - Создание dirty задач для истории

## Описание

Скрипт для создания `StatsDirty` задач для исторических данных. Используется для первоначального заполнения очереди пересчёта статистики.

## Расположение

`server/scripts/backfill_stats_dirty.js`

## Использование

### Базовое использование (только system)

Создать задачи для последних 12 месяцев (только system/day):

```bash
node scripts/backfill_stats_dirty.js --months=12
```

Создать задачи для конкретного диапазона дат:

```bash
node scripts/backfill_stats_dirty.js --start=2025-01-01 --end=2025-12-31
```

### С включением customers

**⚠️ ВНИМАНИЕ:** Создание задач для всех customers может быть очень тяжёлым (например, 100 customers × 365 дней = 36,500 задач). Используйте только при необходимости.

```bash
node scripts/backfill_stats_dirty.js --months=12 --include-customers
```

### С включением carriers и users

```bash
node scripts/backfill_stats_dirty.js --months=12 --include-carriers --include-users
```

### Dry run (проверка без создания)

```bash
node scripts/backfill_stats_dirty.js --months=12 --dry-run
```

## Параметры

### Обязательные

- `--months=N` - Количество месяцев назад от текущей даты
- ИЛИ `--start=YYYY-MM-DD --end=YYYY-MM-DD` - Конкретный диапазон дат

### Опциональные

- `--dry-run` - Режим проверки без создания задач
- `--include-customers` - Включить создание задач для всех активных customers
- `--include-carriers` - Включить создание задач для всех активных carriers
- `--include-users` - Включить создание задач для всех активных users

## Примеры

### Пример 1: Backfill за последний год (только system)

```bash
node scripts/backfill_stats_dirty.js --months=12
```

Создаст ~365 задач для system/day за последний год.

### Пример 2: Backfill за конкретный период с проверкой

```bash
node scripts/backfill_stats_dirty.js --start=2025-01-01 --end=2025-03-31 --dry-run
```

Покажет сколько задач будет создано без реального создания.

### Пример 3: Backfill для активных customers (тяжёлая операция)

```bash
node scripts/backfill_stats_dirty.js --months=6 --include-customers
```

Создаст задачи для всех активных customers за последние 6 месяцев.

## Рекомендации

### Оптимальная стратегия

1. **System/day всегда** - создавать для всех исторических периодов
2. **Customer/day по запросу** - создавать только когда:
   - Пользователь открывает customer dashboard (on-demand)
   - По расписанию ночью для активных customers (например, последние 30 дней)
3. **Carrier/day и User/day** - аналогично customer, по запросу или по расписанию

### Производительность

- **System tasks**: ~365 задач за год (быстро)
- **Customer tasks**: ~36,500 задач за год при 100 customers (медленно)
- Скрипт обрабатывает задачи батчами по 1000 для эффективности
- Использует `upsert` для избежания дубликатов

### Мониторинг

После создания задач воркер (`statsWorker`) автоматически начнёт их обрабатывать. Проверить статус:

```javascript
// Количество незаблокированных задач
StatsDirty.countDocuments({ 'lock.locked': false })

// Количество задач с ошибками
StatsDirty.countDocuments({ attempts: { $gte: 3 } })
```

## Важные замечания

1. **Уникальность**: Скрипт использует `upsert` с уникальным ключом `(grain, dateKey, entityType, entityId)`, поэтому повторный запуск безопасен - дубликаты не создадутся.

2. **Активные сущности**: Для customers/carriers/users создаются задачи только для активных (`status: 'active'`).

3. **Sources**: По умолчанию все задачи создаются с `sources: ['loads', 'receivable', 'payable']` для полного пересчёта.

4. **Приоритет**: Все задачи создаются с `priority: 0`. Можно вручную изменить приоритет для важных периодов.

5. **Lock**: Все задачи создаются разблокированными (`lock.locked: false`) и готовыми к обработке воркером.

## Troubleshooting

### Ошибка: "MongoServerError: E11000 duplicate key error"

Это нормально - означает, что задача уже существует. Скрипт использует `upsert`, поэтому дубликаты пропускаются.

### Слишком много задач создаётся

Если создаётся слишком много задач (например, для всех customers), рассмотрите:
- Ограничение диапазона дат (например, только последний месяц)
- Создание задач только для активных/важных customers
- Использование cron для постепенного создания задач

### Воркер не обрабатывает задачи

Проверьте:
- Запущен ли воркер (`statsWorkerCron`)
- Есть ли незаблокированные задачи в `StatsDirty`
- Логи воркера на наличие ошибок
